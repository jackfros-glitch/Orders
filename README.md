# OrderProcessingApi 

This is a Asp-based backend system for processing customers Orders. It is Built with Scalabilty in mind to handle high traffics spikes. It is also build with Maintainability in Mind as specifiied in the Accessment docs

## Getting Started

## Architecture Decisions
### Idempotency — Duplicate Payment Prevention
To prevent duplicate payments, a payment reference is generated upfront before any payment is made. When a payment request comes in, the system first checks if a payment already exists for that reference. If the payment is found and marked as Paid, the same result is returned without triggering a new payment. If no payment is found, or a previous attempt failed, the payment flow is triggered fresh.
<br/>
### Atomic Updates — Product Stock Management: 
To prevent race conditions during order placement, product stock updates are handled using atomic database updates. Rather than reading stock, validating, and writing back in separate steps, the decrement and validation happen in a single database operation — eliminating the window where concurrent requests could oversell a product.
<br/>
### Transactional Order Placement:
The entire order placement flow is wrapped in a database transaction. If anything fails at any point during the process stock updates, order status changes, or anything else — all changes are rolled back, ensuring the database never ends up in an inconsistent state.
<br/>
## Trade-offs
- The architectural decisions above are intentional and their trade-offs worth noting.
Row-level locking is a common approach to handling race conditions and provides strong consistency guarantees. However it introduces a performance bottleneck that becomes increasingly apparent under high traffic, as concurrent requests queue up waiting for locks to be released. Atomic updates avoid this bottleneck entirely by pushing the conflict check into the database operation itself, making it the more scalable choice.
<br/>
- Request queuing is another valid strategy for handling concurrency and also helps protect the database from traffic spikes. However it works best in scenarios where an immediate response to the user is not required. Additionally, under extremely high traffic, the queue itself can become a bottleneck if not carefully designed and scaled.
